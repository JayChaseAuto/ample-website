## Critical (4)

### C1 — Wildcard CORS lets any browser tab overwrite local source
- **Location**: [`dev-server.py:107`](dev-server.py) (`_cors` returns `Access-Control-Allow-Origin: *` on every response, including `PUT /assets/*` and `POST /__editor/save_index`)
- **Confidence**: 100
- **Failure mode**: While `py dev-server.py` is running, any tab open to `evil.com` can hit `http://127.0.0.1:8000/__editor/save_index` and overwrite `project/index.html`'s EDITMODE block — or `PUT /assets/x.png` to drop attacker-controlled bytes into `project/assets/`. Loopback-only bind blocks shells on other machines but not browsers; the CORS preflight succeeds, the request succeeds, and your next `git push` ships the payload to GitHub Pages.
- **Fix sketch**: In `_cors`, only echo `Origin` when it matches `http://127.0.0.1:<PORT>` / `http://localhost:<PORT>`; reject cross-origin requests on PUT/POST.

### C2 — SVG drag-drop writes script-bearing file into `assets/`
- **Location**: [`project/src/Animations.jsx:241`](project/src/Animations.jsx) (canvas re-encode skipped for `image/svg+xml`) → [`dev-server.py:50`](dev-server.py) (`SAFE_NAME_RE` allows `svg`)
- **Confidence**: 80
- **Failure mode**: Drag any `<svg onload="...">` (or a remote SVG that passes CORS) onto a drop slot. `writeImageToAssets` ships the raw SVG bytes via PUT; the server stores them as `assets/<prefix>-<hash>.svg`, Save & lock bakes that path into source. The SVG is harmless inside `<img>`/`url()`, but the file lives in the committed repo — any direct navigation to `https://you.github.io/.../assets/...svg` (search-engine indexing, hot-linking from another tab) executes the embedded `<script>` in the GitHub Pages origin.
- **Fix sketch**: Reject `image/svg+xml` blobs in `writeImageToAssets`, **or** strip `svg` from `SAFE_NAME_RE`'s allowlist.

### C3 — Auto-reload poll races image uploads → silent data loss
- **Location**: [`project/index.html:1591-1594`](project/index.html), with [`dev-server.py:60-80`](dev-server.py) bumping mtime on save
- **Confidence**: 85
- **Failure mode**: The 2s `setInterval` reloads on any version change with no guard for `uploads.inFlight > 0`. `assets/` is excluded from the watch (so a bare drop is safe), but `Save & lock` rewrites `index.html` and bumps mtime. If the user drops an image and immediately clicks Save (or just saves while a slow upload is still PUTting), the reload fires while `writeImageToAssets` is mid-fetch — the asset never lands on disk, the freshly-saved JSON references an absent path. The existing comment ("assets/ excluded so drops don't trigger reload") only covers drops in isolation, not the save-while-uploading combination.
- **Fix sketch**: In `tick()`, expose `window.__ampleUploadsInFlight` from App and skip `location.reload()` while `> 0`; also gate during `saveResult?.pending`.

### C4 — `exportTweaksState` uses bare `lift()` for slot-shaped fields → blob URLs ship into source
- **Location**: [`project/index.html:441, 451, 454, 468`](project/index.html) (`heroImage`, `productOverrides[].heroBgImage`, `productOverrides[].calloutImage`, `goldStandardMedia`)
- **Confidence**: 80
- **Failure mode**: `liftSlot` correctly handles both bare strings and `{ url, position, scale, fit }` objects. It's wired into `categoryImages`, `catalogCardImages`, `storyImages`, `goldFeatureImages`. But `heroImage`, `productOverrides[slug].heroBgImage`, `productOverrides[slug].calloutImage`, and `goldStandardMedia` still use `lift()` directly. `needsUpload(value)` returns `false` for an object, so a blob: / data: URL nested in `.url` is shipped to source unchanged. Future you adds per-slot knobs for any of those fields → silent save corruption.
- **Fix sketch**: Route all image-bearing fields through `liftSlot`, not `lift`, even for single-image keys.

## High (4)

### H1 — `useTweaks` returns a new `history` object every render
- **Location**: [`project/src/tweaks-panel.jsx:371`](project/src/tweaks-panel.jsx) → consumed at [`project/index.html:619, 670, 693`](project/index.html)
- **Confidence**: 95
- **Why**: `return [values, setTweak, { undo, redo, canUndo, canRedo }];` builds a new object literal every render. In `App`, `history` is the 6th dep of `tweaksContextValue`'s `useMemo`, so the context value is recreated every render. Cascade: (a) every `useTweakState()` consumer (every `FeaturedCard`, `CategoryCard`, callout, gold feature, story card) re-renders on every App render; (b) `TweaksProvider`'s `useEffect([value])` re-fires and re-writes four `window.__ample*` mirrors every render; (c) the global Ctrl+Z keydown listener detaches/re-attaches every render. Real perf bug, not a nit.
- **Fix sketch**: `const history = React.useMemo(() => ({ undo, redo, canUndo, canRedo }), [undo, redo, canUndo, canRedo]);` and return that.

### H2 — Undo after an image drop restores a revoked blob: URL
- **Location**: [`project/src/Animations.jsx:351-362`](project/src/Animations.jsx)
- **Confidence**: 90
- **Why**: `useImageDrop` calls `onDrop(blobUrl)` (which `setTweak`s and pushes pre-drop state to history), `await writeImageToAssets`, then `onDrop(path)` (a SECOND `setTweak` that pushes the blob-URL state to history because the upload took well over the 350 ms debounce), then `URL.revokeObjectURL(previewUrl)`. Press Ctrl+Z after the upload settles → state restores to the snapshot containing `blobUrl`, but the object URL was already revoked → broken image, no recovery.
- **Fix sketch**: After the upload succeeds, post-process the most recent history entry to swap blob: URLs for the canonical `assets/...` path. Alternatively, add a `setTweak` option that bypasses the history push for the path-swap call.

### H3 — Editor slider shows `2.5` but runtime clamps to `1.6` → silent downgrade on first nudge
- **Location**: [`project/src/ProductDetailPage.jsx:38-40`](project/src/ProductDetailPage.jsx) (`Math.max(0.5, Math.min(1.6, raw))`), [`project/index.html:1255-1257`](project/index.html) (slider `max=1.6`), persisted value of `2.5` at [`project/index.html:209`](project/index.html) for `caliper-npc`
- **Confidence**: 90
- **Why**: The panel's "Image scale" slider reads the raw `productData.calloutImageScale` so it displays `2.5` for caliper-npc, but the rendered page applies the clamp and shows `1.6`. Editor and live page disagree on what the same setting means. Worse, the slider caps user input at `1.6`, so the moment the user nudges the slider the persisted `2.5` is overwritten — silently downgraded forever.
- **Fix sketch**: Pick one range and enforce it on both sides. Either widen runtime + slider to `[0.5, 2.5]`, or clamp on write so the persisted value never exceeds the slider max.

### H4 — Dragged `<img>` URLs stored verbatim → leak to GitHub Pages
- **Location**: [`project/src/Animations.jsx:207-220`](project/src/Animations.jsx) (`text/html` + `text/uri-list` branches) → line 346 (`onDropRef.current(resolved.url)`)
- **Confidence**: 80
- **Failure mode**: When CORS or non-HTTPS makes `fetchAsMedia` throw, `resolveDraggedToBlob` falls through to `{ url: tryUrls[0] }` with no scheme/host validation. Only `text/plain` has a tiny allowlist; `text/uri-list` and the `<img src>` regex don't. Drag any `<img>` from a third-party site → its URL persists into `tweaks.heroImage` / `catalogCardImages[...]`, and Save & lock bakes a verbatim `https://evil.com/track.png` into the committed `index.html`. Every GitHub Pages visitor then pulls that asset; the third party can swap contents at will, and the URL leaks visitor IP/UA on every page load.
- **Fix sketch**: Drop the `{ url: ... }` fallback entirely (only persist blobs), or restrict bare-URL storage to same-origin / `assets/` prefixes.

## Medium (4)

### M1 — Dead `window.__ample*` setter shims
- **Location**: [`project/src/tweaks-panel.jsx:432-436`](project/src/tweaks-panel.jsx) (TweaksProvider effect writes `__ampleSetTweak`, `__ampleSetProductTweak`, `__ampleSetCategoryImage`)
- **Confidence**: 90
- **Why**: Framed as "compat shims for legacy callers"; a repo-wide grep finds zero readers. Only `window.__ampleTweaks` (the value, not the setters) is read once at `index.html:495`. Every render writes four globals nobody reads, and any consumer that *did* read them would get the previous render's setter for one frame (stale-state footgun).
- **Fix sketch**: Drop the three setter assignments. Keep `window.__ampleTweaks = value.tweaks` since `index.html` reads it.

### M2 — `CategoryCard` reads `tweaks` from prop AND context
- **Location**: [`project/src/HomePage.jsx:86, 265-271`](project/src/HomePage.jsx)
- **Confidence**: 80
- **Why**: Every other card (`FeaturedCard`, `CatalogCard`, `RelatedProductCard`, `LabMediaBox`, gold/story) pulls tweaks from `useTweakState()` directly. `CategoryCard` is the lone holdout: parent passes `tweaks={tweaks}` as a prop AND the child also calls `useTweakState()` for `mergeImageSlot`. Two paths to the same state, with subtly different update guarantees depending on parent memoization.
- **Fix sketch**: Drop the prop on both sides; have `CategoryCard` destructure `tweaks` from `useTweakState()` like its siblings.

### M3 — `catalogCardImages` stays string-only while siblings became objects
- **Location**: [`project/src/tweaks-panel.jsx:378-396`](project/src/tweaks-panel.jsx) (`readImageSlot`), [`project/index.html:1168`](project/index.html) (`mergeImageBag` for catalogCardImages) vs. [`project/index.html:1108, 1279, 1306`](project/index.html) (categoryImages / goldFeatureImages / storyImages all use `mergeImageSlot`)
- **Confidence**: 80
- **Why**: Phase 5 upgraded `categoryImages` / `storyImages` / `goldFeatureImages` to `{url, position, scale, fit}` slot objects via `mergeImageSlot`. `catalogCardImages` was left as plain strings; its position/scale/padding live in parallel fields under `productOverrides[slug]`. Two storage schemes for the same conceptual thing. Editor logic can't be shared, and `mergeImageBag` exists solely to feed this one bag.
- **Fix sketch**: Migrate `catalogCardImages` to slot form and retire `mergeImageBag` — or document the dual scheme prominently so it doesn't look like an accident.

### M4 — Error overlay grows `textContent` unbounded on fault loops
- **Location**: [`project/index.html:41`](project/index.html) (`box.textContent += '[' + kind + '] ' + msg + '\n\n';`)
- **Confidence**: 80
- **Failure mode**: Any recurring error (a `setInterval` callback that throws, the auto-reload tick on malformed JSON, an `unhandledrejection` in render) keeps appending. No cap, no dedupe, no session clear. A render-loop error firing at ~60 Hz adds ~1 KB/frame, exhausts heap in under a minute, and bricks the tab — precisely when the developer needs the overlay to be readable. Editor-only, so the impact is on you mid-edit, with the in-memory tweak state lost.
- **Fix sketch**: Cap `box.textContent.length` (truncate at ~64 KB), or coalesce consecutive identical messages with a count badge.

## Methodology

Adapted from `~/.claude/plugins/marketplaces/claude-plugins-official/plugins/code-review/commands/code-review.md`. Four parallel reviewers, each with a single focus, were told to use codegraph for cross-file lookups (no Grep+Read chains for symbol resolution) and to only report issues at confidence ≥80.

| Reviewer | Returned | After dedup |
|---|---|---|
| Bug scan | 4 issues | 4 |
| Architecture | 6 issues | 5 (one merged with comment-drift) |
| Comment drift | 2 issues | 1 (one merged with architecture) |
| Security | 4 issues | 4 |
| **Total** | 16 | **13** |

Tasks done **outside** the agents (so you can audit the methodology):
- I provided each agent a 4–6 sentence brief on architecture history (Context migration, slot schema, Phase 5) so they didn't have to rediscover it.
- I told them what counts as a "false positive" (lint nits, missing tests, pre-existing issues, theoretical attacks requiring remote network).
- After agents returned, I deduped issues that overlapped (L2 covers both an architecture finding and a comment-drift finding).

## Suggested order of attack

If you're going to triage these in one sitting:

1. **C1** (CORS) and **C2** (SVG XSS) — actual security holes that can ship malicious bytes to your GitHub Pages site. ~15 minutes each.
2. **H1** (history identity) — single-line `useMemo` wrap; cascades into a measurable perf win.
3. **C4** + **C3** — both are silent-data-loss bugs around the save/upload boundary. Same area, fix together.
4. **H2** (undo + revoked blob), **H3** (clamp mismatch) — user-visible bugs.
5. **H4** (URL exfil) — fix by tightening the URL fallback branch.
6. **M1**–**M4** — code health, batch in one cleanup pass.
7. **L1**, **L2** — comment touch-ups, 5 minutes total.
